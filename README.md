# Xiangqi EasyX / 中国象棋

一个基于 **C++17、Visual Studio 2022 和 EasyX 图形库** 编写的图形化中国象棋游戏。项目实现了标准 10×9 中国象棋棋盘、红黑双方 32 枚棋子、鼠标选子落子、合法走法提示、吃子、将军检测、胜负判断、悔棋、开局计时规则选择、超时判负、局势分析进度条、UTF-8 日志记录，以及基于 **Pikafish 2026-01-02 + NNUE** 的高质量机器人对战功能。

**v3.6** 在 v2.2 的计时和局势分析基础上，新增了完整的机器人对战系统。玩家可以选择双方玩家、红方机器人、黑方机器人或双方机器人四种模式。机器人通过 Windows 管道与 Pikafish 象棋引擎进行 UCI 通信，在后台线程搜索走法，并在执行前再次调用本项目规则层校验合法性，避免引擎输出与本地规则不一致。

## 2. 项目简介

本项目实现了一个本地运行的图形化中国象棋程序。游戏启动后，玩家首先进入统一配置界面，可以选择每步时长、单方总时长和对战模式。确认设置后，程序进入标准棋局，红方先手，黑方后手。

真人玩家通过鼠标点击己方棋子进行选择，程序自动计算该棋子的所有合法落点，并用绿色标记显示。点击合法落点即可完成移动；如果目标位置存在敌方棋子，则执行吃子。

项目实现了中国象棋七类棋子的主要走法，并加入“不能送将”的合法性过滤。程序会先根据棋子类型生成候选走法，再复制当前局面模拟走棋，判断走完后本方将帅是否仍会被攻击。会导致本方被将军的走法不会显示，也不能执行。

v3.6 新增机器人对战功能。机器人不是通过简单随机走棋或固定权值表选择动作，而是调用项目自带的 Pikafish 高质量象棋引擎。程序将当前历史走法转换为 UCI `position startpos moves` 命令，要求引擎在限定时间内搜索，并解析 `info`、`score`、`pv` 和 `bestmove` 等输出。引擎给出的走法转换为项目内部坐标后，还会通过 `getlegalmoves()` 再次验证，确保最终走法符合本项目棋盘规则。

为了避免引擎搜索卡住 EasyX 主循环，搜索任务通过 `std::async` 放在后台线程中执行。主线程继续刷新计时、局势分析和界面。搜索完成后，机器人先在界面中选中棋子，短暂等待，再执行落子，使自动对局过程清晰可见。

项目结构将棋盘规则、棋子数据、界面绘制、日志记录、计时系统、局势分析和机器人引擎通信分别实现，便于阅读、调试、答辩和二次开发。

## 3. 已实现功能

### 3.1 基础对局功能

- 标准中国象棋 10 行 × 9 列棋盘。
- 开局自动初始化 32 枚棋子。
- 红方先手，黑方后手。
- 支持鼠标点击选择当前方棋子。
- 支持自动显示当前棋子的全部合法落点。
- 支持普通移动。
- 支持吃子。
- 支持双方轮流行棋。
- 支持吃掉将帅后结束游戏。
- 支持一方无合法走法时判负。
- 支持一方每步时间或总时间耗尽时判负。
- 支持游戏结束后显示获胜方。
- 支持按 `Esc` 退出窗口。

### 3.2 开局配置界面

v3.6 将原有计时选择界面扩展为统一游戏配置界面，可同时设置：

| 设置项 | 可选值 |
| --- | --- |
| 每步时长 | 30 秒、60 秒、90 秒 |
| 单方总时长 | 5 分钟、10 分钟、15 分钟、30 分钟 |
| 对战模式 | 双方玩家、红方机器人、黑方机器人、双方机器人 |

默认设置为：

```text
每步时长：60 秒
单方总时长：10 分钟
对战模式：双方玩家
```

设置界面支持鼠标和键盘操作：

```text
↑ / ↓：切换“每步时长、单方总时长、对战模式”
← / →：调整当前选项
Enter：开始游戏
Esc：退出程序
```

当前键盘焦点使用红色外框标记，已选中的选项使用绿色按钮显示。

### 3.3 对战模式

当前版本提供四种模式：

1. **双方玩家**  
   红方和黑方均由真人通过鼠标操作。

2. **红方机器人**  
   红方由 Pikafish 控制，黑方由真人操作。

3. **黑方机器人**  
   红方由真人操作，黑方由 Pikafish 控制。

4. **双方机器人**  
   红黑双方均由 Pikafish 控制，可以自动完成整局对弈。

当当前行动方是机器人，或机器人正在搜索、选中和落子时，程序会阻止真人点击棋盘或悔棋，避免人为操作与自动走棋发生冲突。`Esc` 仍可用于退出程序。

### 3.4 Pikafish 高质量机器人

- 项目内置 Pikafish 2026-01-02 Windows x64 引擎。
- 使用 `pikafish.nnue` 完成神经网络局面评估。
- 通过 UCI 协议与引擎通信。
- 使用 Windows 匿名管道传递标准输入和标准输出。
- 使用 `CreateProcessW()` 创建无控制台窗口的引擎子进程。
- 使用 `std::async` 在后台执行搜索。
- 主线程通过非阻塞轮询检查搜索是否完成。
- 搜索期间 EasyX 窗口继续刷新。
- 根据电脑逻辑处理器数量自动配置搜索线程。
- 引擎线程数最多为 8。
- 引擎 Hash 设置为 256 MB。
- 引擎启动时自动加载 NNUE 文件。
- 引擎输出走法会再次通过本项目合法走法函数校验。
- 引擎无法启动、搜索失败或走法不合法时，会显示错误对话框并记录日志。
- 程序退出时会发送 `quit` 命令并释放进程、线程和管道句柄。

### 3.5 双机器人平衡开局

双方机器人模式下，为减少每次对局完全重复，前 16 个半回合启用平衡开局变化：

- Pikafish 使用 `MultiPV = 3` 输出多个高质量候选。
- 只保留与第一候选评分差距不超过 8 的走法。
- 在这些高质量候选中按评分权重随机选择。
- 分数越接近第一候选，获得的权重越高。
- 如果只有一个满足条件的候选，则仍采用第一推荐。
- 搜索进入中后盘后恢复 `MultiPV = 1`，直接选择最佳走法。
- 出现接近将死的极端评分时，不再随机选择。

该机制只用于双方机器人开局变化，不会让机器人故意选择明显较差的走法。

### 3.6 机器人搜索时间与可视化

机器人搜索时间会根据当前每步时长自动调整：

| 每步时长 | 首选引擎搜索时间 |
| --- | ---: |
| 30 秒 | 约 4.2 秒 |
| 60 秒 | 约 7 秒 |
| 90 秒 | 约 9 秒 |

程序还会计算搜索时间上限，使机器人从开始搜索到最终落子的整体过程原则上不超过当前每步时长的约 30%。

机器人走棋分为三个可视化阶段：

```text
后台搜索
    ↓
选中引擎指定棋子
    ↓
短暂等待
    ↓
执行目标落点
```

等待时间根据每步时长动态计算，单段通常不超过 2 秒。

### 3.7 对局计时功能

- 右侧面板实时显示本步剩余时间。
- 右侧面板分别显示红方、黑方总剩余时间。
- 当前行动方走棋时，同时扣减本步时长和该方总时长。
- 机器人搜索和等待过程同样计入机器人一方时间。
- 成功走棋后，自动切换到下一方计时。
- 本步时间耗尽时，当前方超时判负。
- 单方总时间耗尽时，该方超时判负。
- 游戏结束后计时停止。
- 悔棋时恢复上一手之前的计时状态。
- 机器人模式下，如果引擎搜索超过玩家剩余时间，计时系统仍以实际经过时间为准判定。

### 3.8 棋子规则

当前版本实现中国象棋七类棋子的主要走法：

| 棋子 | 规则实现 |
| --- | --- |
| 帅 / 将 | 只能在九宫内上下左右移动一步，并支持将帅照面 |
| 仕 / 士 | 只能在九宫内斜走一步 |
| 相 / 象 | 走田字，不能过河，并处理“塞象眼” |
| 马 | 走日字，并处理“蹩马腿” |
| 车 | 横竖直线移动，路径中不能有棋子阻挡 |
| 炮 | 不吃子时直线移动；吃子时必须隔一个棋子作为炮架 |
| 兵 / 卒 | 过河前只能向前一步；过河后可向前或左右移动，不能后退 |

### 3.9 将军与胜负判断

- 支持判断某一方将帅是否被攻击。
- 支持将帅照面检测。
- 支持炮隔一子攻击检测。
- 支持车直线攻击检测。
- 支持马、象、士、兵卒攻击范围检测。
- 支持界面显示“将军”提示。
- 支持箭头标明进攻方和被将军方。
- 每次走棋后判断下一方是否还有合法走法。
- 吃掉对方将帅后游戏直接结束。
- 对方没有任何合法走法时，当前走棋方获胜。
- 一方超时时，另一方直接获胜。
- 引擎异常导致机器人无法继续走棋时，机器人一方判负。

### 3.10 合法走法提示

玩家点击当前方棋子后，程序自动计算全部合法落点：

- 绿色实心圆点表示可以移动到的空位。
- 绿色圆环表示可以吃掉的敌方棋子。
- 当前选中棋子显示绿色外框。
- 己方棋子占据的位置不会作为落点。
- 会导致本方将帅被攻击的位置不会作为落点。
- 游戏结束后不能继续选择和移动棋子。

机器人从 UCI 走法转换得到内部落点后，也使用同一套合法性判断，不会绕过本地规则层。

### 3.11 局势分析功能

底部局势分析条根据当前棋盘局面计算红黑双方综合分，并转换成百分比显示：

- 红方局势百分比。
- 黑方局势百分比。
- 红黑双色进度条。
- 真人或机器人走棋后自动重新计算。
- 悔棋后自动恢复并重新计算。

该局势分析仍然是项目内置的启发式评估，不直接显示 Pikafish 的 NNUE 分数。它综合考虑：

- 棋子基础价值；
- 兵卒是否过河；
- 棋子是否受到己方保护；
- 棋子之间是否形成配合；
- 是否威胁对方将帅；
- 将帅是否受困；
- 双方剩余棋子情况。

因此，底部百分比主要用于课程项目展示，并不等同于专业引擎给出的严格胜率。

### 3.12 悔棋功能

- 点击右侧“撤回上一步”按钮可以悔棋。
- 按键盘 `U` 可以悔棋。
- 悔棋会恢复移动棋子位置。
- 如果上一手发生吃子，会恢复被吃棋子。
- 悔棋后恢复到上一手行动方。
- 游戏结束后也可以悔棋。
- 悔棋会同步恢复计时状态。
- 悔棋后重新计算局势分析。
- 悔棋后会取消尚未结束的机器人后台搜索。
- 当前机器人正在搜索、选中或落子时，真人不能悔棋。
- 悔棋成功后，若恢复到机器人一方，机器人会根据恢复后的局面重新搜索。

### 3.13 图形界面功能

- 使用 EasyX 创建 980 × 760 图形窗口。
- 左侧绘制棋盘、九宫、楚河汉界和棋子。
- 右侧绘制当前玩家、计时信息、悔棋按钮、双方棋子列表和操作提示。
- 底部绘制红黑双方局势分析进度条。
- 设置界面增加对战模式选项。
- 当前设置组支持键盘焦点提示。
- 使用不同颜色区分红方和黑方棋子。
- 被吃棋子在右侧列表中用红色叉号标记。
- 棋盘使用圆形棋子和中文棋子文字。
- 使用 `Microsoft YaHei` 和 `SimSun` 字体绘制中文。
- 使用 `BeginBatchDraw()` 和 `EndBatchDraw()` 减少闪烁。
- 主循环约每 180 毫秒刷新计时显示。
- 机器人选中棋子后会显示与真人相同的绿色选中外框。
- 机器人搜索通过后台线程执行，不会长时间冻结窗口。

### 3.14 日志系统

- 程序启动时自动创建日志文件。
- 日志目录为工程目录下的 `result/`。
- 程序优先向上查找 `xiangqi_easyx.vcxproj` 所在目录。
- 日志文件名格式为 `YYYYMMDD_HHMMSS.txt`。
- 日志使用 UTF-8 with BOM 编码。
- 日志记录游戏开始。
- 日志记录计时规则。
- 日志记录对战模式。
- 日志记录真人和机器人每一步移动。
- 日志记录吃子。
- 日志记录悔棋。
- 日志记录超时。
- 日志记录游戏结束。
- 日志记录引擎启动或搜索错误。
- 日志记录窗口关闭。

## 4. 运行环境

运行本项目需要以下环境：

1. Windows 10 或更新版本；
2. Visual Studio 2022；
3. 已安装“使用 C++ 的桌面开发”组件；
4. 已安装 EasyX 图形库；
5. 支持 C++17 的 MSVC 编译器；
6. 支持 SSE4.1 和 POPCNT 指令集的 x86-64 处理器；
7. 工程目录中保留完整 `engine/` 文件夹。

项目使用 EasyX 的：

```text
graphics.h
initgraph()
ExMessage
peekmessage()
getmessage()
BeginBatchDraw()
EndBatchDraw()
```

机器人模块使用 Windows API：

```text
CreateProcessW()
CreatePipe()
WriteFile()
ReadFile()
PeekNamedPipe()
WaitForSingleObject()
TerminateProcess()
```

工程文件中的平台工具集为：

```text
v145
```

如果本地 Visual Studio 没有对应工具集，可以在项目属性中修改为本机已安装的 MSVC 工具集。

工程字符集为 Unicode，编译参数包含：

```text
/utf-8
```

虽然工程同时保留 Win32 和 x64 配置，但项目自带的 `pikafish.exe` 是 x86-64 程序，因此使用机器人功能时推荐选择：

```text
Debug | x64
Release | x64
```

## 5. 运行方法

1. 下载或克隆本项目到本地。
2. 使用 Visual Studio 2022 打开：

```text
xiangqi_easyx.sln
```

3. 选择：

```text
Debug | x64
```

或：

```text
Release | x64
```

4. 确认 EasyX 已正确安装。
5. 确认以下文件存在：

```text
xiangqi_easyx/
└── engine/
    ├── pikafish.exe
    ├── pikafish.nnue
    ├── Copying.txt
    ├── NNUE-License.md
    ├── SOURCE.txt
    └── AUTHORS
```

6. 点击“本地 Windows 调试器”运行。
7. 在配置界面选择：
   - 每步时长；
   - 单方总时长；
   - 对战模式。
8. 点击“开始游戏”，或按 `Enter`。
9. 真人回合：
   - 点击己方棋子；
   - 点击绿色落点完成走棋；
   - 点击“撤回上一步”或按 `U` 悔棋。
10. 机器人回合：
    - 等待后台搜索完成；
    - 程序会自动选中棋子并完成落子。
11. 按 `Esc` 退出。

如果选择机器人模式但引擎文件缺失、CPU 不支持对应指令集或 UCI 初始化失败，程序会显示错误窗口并退出。

## 9. 主要算法与实现思路

### 9.1 游戏主流程

```text
创建 EasyX 窗口
      ↓
选择每步时长、总时长和对战模式
      ↓
若有机器人则初始化 Pikafish
      ↓
初始化棋盘、计时、分析和日志
      ↓
进入非阻塞消息循环
      ↓
刷新倒计时
      ↓
推进机器人搜索或可视化阶段
      ↓
处理真人鼠标和键盘输入
      ↓
执行走棋、切换计时、重新分析
      ↓
判断胜负或超时
```

### 9.2 基础走法生成

程序根据棋子类型调用不同的候选走法函数。基础走法只处理棋子自身规则，不处理走完以后本方是否被将军。

### 9.3 九宫判断

将帅和士的目标位置需要满足：

```text
红方九宫：第 8–10 行，第 4–6 列
黑方九宫：第 1–3 行，第 4–6 列
```

程序使用统一的九宫判断函数限制其活动范围。

### 9.4 相/象不能过河与塞象眼

相/象每次移动两个斜格。程序先检查目标是否仍在本方半场，再计算中间象眼位置。象眼有棋子时，该走法无效。

### 9.5 马的蹩马腿

程序先判断八个“日”字目标，再根据移动方向计算马腿位置。马腿被占据时，不加入候选落点。

### 9.6 车的直线扫描

车沿四个方向逐格扫描：

- 遇到空位：加入落点并继续；
- 遇到敌方棋子：加入吃子落点并停止；
- 遇到己方棋子：直接停止。

### 9.7 炮的炮架规则

炮的扫描分为两段：

- 未遇炮架前，空位可以移动；
- 遇到第一个棋子后，该棋子作为炮架；
- 炮架之后遇到的第一个敌方棋子可以吃；
- 遇到第二个棋子后停止扫描。

### 9.8 兵/卒过河判断

兵卒始终只能向前。过河后额外允许左右移动，仍然不能后退。

### 9.9 将帅照面

将和帅位于同一列且中间没有棋子时，彼此形成直接攻击。程序既在攻击检测中处理该规则，也会通过合法性过滤阻止走出将帅照面的局面。

### 9.10 攻击检测

`pieceattackssquare()` 根据攻击者类型判断其是否正在攻击指定位置。该函数用于将军判断，不直接等同于完整走法生成。

### 9.11 将军判断

程序查找指定阵营仍存活的将帅，再遍历对方全部棋子。只要有一枚棋子攻击将帅位置，就判定该方被将军。

### 9.12 合法走法过滤

完整合法走法生成流程：

```text
生成基础候选走法
      ↓
复制当前局面
      ↓
模拟执行候选走法
      ↓
判断本方将帅是否被攻击
      ↓
未被攻击：保留
仍被攻击：删除
```

### 9.13 真实走棋

执行真实走棋时：

1. 检查棋子是否属于当前方；
2. 检查目标是否在合法走法中；
3. 记录 `moveinfo`；
4. 保存历史；
5. 移动棋子；
6. 处理吃子；
7. 判断是否吃掉将帅；
8. 切换当前方；
9. 判断下一方是否还有合法走法；
10. 更新计时、日志和局势分析。

### 9.14 悔棋实现

悔棋取出最近一条 `moveinfo`：

- 将移动棋子放回起点；
- 恢复被吃棋子；
- 恢复当前行动方；
- 清除游戏结束状态；
- 恢复计时快照；
- 重新计算分析；
- 取消正在运行的机器人后台搜索。

### 9.15 计时系统

计时器通过 `GetTickCount()` 获取真实毫秒数。每次刷新同时扣减：

- 当前步骤剩余时间；
- 当前一方总剩余时间。

走棋完成后保存计时快照并切换行动方。机器人搜索、选择等待和落子等待都会被计入实际时间。

### 9.16 局势分析算法

局势分析使用本地固定权值模型计算：

```text
基础棋子价值
+ 过河奖励
+ 保护奖励
+ 配合奖励
+ 将军威胁
- 受困惩罚
```

红黑得分转换成总和为 100% 的百分比，用于绘制底部进度条。

### 9.17 UCI 局面转换

程序根据 `state.history` 生成：

```text
position startpos moves <走法1> <走法2> ...
```

棋盘内部坐标转换为 UCI 坐标：

```text
列号 0–8 → a–i
内部行号 → 9–0
```

例如内部起点和终点被组合为四字符 UCI 走法。

### 9.18 Pikafish 初始化

引擎初始化流程：

```text
检查 pikafish.exe 和 pikafish.nnue
      ↓
创建输入输出管道
      ↓
CreateProcessW 启动引擎
      ↓
发送 uci
      ↓
等待 uciok
      ↓
设置 Threads、Hash、EvalFile、MultiPV
      ↓
发送 isready
      ↓
等待 readyok
      ↓
发送 ucinewgame
```

### 9.19 后台搜索

主线程复制当前 `gamestate`，使用：

```cpp
std::async(std::launch::async, ...)
```

启动搜索任务。后台线程向 Pikafish 发送：

```text
go movetime <毫秒数>
```

主线程使用 `future.wait_for(0ms)` 非阻塞检查结果，因此窗口、计时和输入循环不会因引擎搜索完全停止。

### 9.20 引擎输出解析

程序解析：

- `depth`：搜索深度；
- `multipv`：候选排名；
- `score cp`：普通局面评分；
- `score mate`：将死评分；
- `pv`：主要变化；
- `bestmove`：最终推荐走法。

将死评分会转换为接近正负 100000 的内部数值。

### 9.21 双机器人开局变化

双方机器人前 16 个半回合使用 `MultiPV = 3`。在与最优候选分差不超过 8 的走法中，按照：

```text
weight = exp((candidate_score - best_score) / 3)
```

计算权重并随机选择，从而在保持高质量的同时减少重复开局。

### 9.22 机器人走法二次校验

Pikafish 走法转换为内部行列后：

1. 检查起点、终点是否在棋盘内；
2. 检查起点是否存在当前方棋子；
3. 调用 `getlegalmoves()`；
4. 判断引擎目标是否存在于合法落点列表；
5. 只有通过校验才允许执行。

### 9.23 引擎取消与资源释放

悔棋或程序退出时：

- 设置取消标记；
- 如果引擎仍在搜索，发送 `stop`；
- 等待后台任务结束；
- 发送 `quit`；
- 等待引擎正常退出；
- 超时则强制结束进程；
- 关闭输入管道、输出管道、线程和进程句柄。

### 9.24 日志输出

日志模块使用 UTF-8 with BOM 保存中文内容。每条日志包含时间顺序信息，并记录真人和机器人使用同一格式完成的走法。

## 12. 未完成内容与后续可优化方向

1. **增加机器人难度等级**  
   当前根据计时设置分配搜索时间，没有单独的简单、普通、困难选项。

2. **增加固定搜索深度模式**  
   当前主要使用 `go movetime`，后续可以支持 `go depth`。

3. **显示引擎思考信息**  
   可以在界面中显示搜索深度、局面分、候选走法和主要变化。

4. **将底部分析条接入 Pikafish**  
   当前分析条仍使用本地启发式模型，与机器人 NNUE 分数互相独立。

5. **增加机器人暂停与托管**  
   可以在对局中临时切换真人和机器人控制。

6. **增加重新开始按钮**  
   当前需关闭程序后重新启动新对局。

7. **增加保存与读档**  
   当前只能记录文本日志，不能恢复棋局。

8. **增加标准棋谱导出**  
   当前日志按行列记录，可进一步生成中文纵线记谱或 UCI 棋谱。

9. **完善和棋规则**  
   当前尚未实现长将、长捉、重复局面、自然限着等完整竞赛规则。

10. **完善引擎容错**  
    当前引擎搜索失败时机器人直接判负，后续可增加本地备用走法算法。

11. **支持更多引擎配置**  
    可以将 Threads、Hash、NNUE 文件和搜索时间写入配置文件。

12. **提高跨平台能力**  
    当前 EasyX、Windows 管道和 Pikafish Windows 程序均依赖 Windows。

## 17. 结语

v3.6 的重点不只是增加一个“自动走棋”按钮，而是把专业象棋引擎完整接入原有课程项目。程序需要完成外部进程启动、UCI 命令通信、NNUE 加载、后台线程搜索、搜索结果解析、坐标转换、本地合法性复核、可视化选中、计时同步和资源释放等多个环节。

项目同时保留了自己实现的中国象棋规则层和本地局势分析层。Pikafish 负责提供高质量决策，`board.cpp` 负责确保最终执行的每一步都符合本项目规则，`analysis.cpp` 则继续为界面提供可解释的红黑局势百分比。三者职责相互独立，避免把外部引擎直接当成整个项目的替代品。

通过本项目，可以练习 C++ 数据结构、规则建模、EasyX 图形界面、非阻塞消息循环、Windows 进程与管道、异步任务、UCI 协议、资源管理和第三方开源组件集成。希望本项目能为中国象棋程序、课程设计、桌面游戏和基础游戏 AI 开发提供参考。
